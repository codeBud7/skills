# Growth plan — output headings and rules

Use with [`SKILL.md`](SKILL.md) for evidence gathering, ladder mapping, and gates.

## Output headings

Use short paragraphs under these headings:

`Plan frame`

`Strengths to leverage`

`Gaps vs target level`

`Growth goals`

`Progress signals`

`Open questions`

`Missing context`

`Sources checked`

## Plan frame

Role, current level, target level, track (IC / management), and review/evidence window. Reference `expectations/ladder.md` for target-level criteria summary.

## Strengths to leverage

Evidence-backed strengths that support the target level. Cite sources. Do not invent strengths from thin evidence.

## Gaps vs target level

Compare evidence to target-level criteria from `expectations/ladder.md`. Name gaps the evidence supports; flag where evidence is thin instead of asserting a gap.

Map to ladder dimensions when possible: scope/impact, technical, leadership/collaboration, autonomy.

## Growth goals

2–4 concrete, forward-looking goals. Each goal should:

- Tie to a gap or strength from above
- Be actionable within a reasonable horizon (quarter-scale unless user specifies otherwise)
- Avoid inventing specific projects unless evidence or user input names them

If evidence is too thin, write fewer goals and expand `Open questions` instead.

## Progress signals

For each goal, state how you would know progress — observable outcomes, behaviors, or deliverables. Not vanity metrics.

## Rules

- Evidence binding: goals without cited evidence or user-stated intent get flagged, not fabricated.
- Forward-looking only: this is a development plan, not a performance review.
- No promotion verdicts unless explicitly requested.
- Use lightweight source tags like `[meeting]`, `[Linear]`, `[GitHub]`, `[ladder]`, `[profile]`. Keep links in `Sources checked`.
